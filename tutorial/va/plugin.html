Plugin System

VA Framework provides an easy inter-component interaction mechanism via the Plugin interface. Developers can either implement the IPlugin interface or inherit Plugin class to create their own plugin.

Only the plugins with the public modifier will be loaded automatically by the Virtual Assistant. To avoid automatic loading of plugins set the modifier to internal and manually load the plugins.

Parts of a Plugin

  • VA gets the VirtualAssistant.Instance
  • Settings gets the Settings instance that belongs to this plugin
  • Name the name of the Plugin
  • SpecialName name used for displaying the Plugin to users
  • GetPanel<T> function called to retrieve a GUI settings panel

Start Plugin

By convention the first plugin that is to be loaded into the VA Framework is the Start plugin. This plugin is supposed to give the VA a functional architecture.

For example Syn Virtual Assistant loads the Syn.VA.Plugins.Start.dll plugin from the root directory which ensures all SIML based plugins are loaded properly into the VA Framework

The General Plugin is NOT placed in the Plugins Directory. It must reside in the root directory of the Virtual Assistant application.

GetPanel()

A Plugin is itself responsible for providing a GUI that enables its user to change the internal values or settings of the plugin. For this, the method GetPanel<T> is called by the Virtual Assistant's Settings Window whenever the Settings Window is opened.

The type T depends on the calling application. The SettingsWindow in Syn Virtual Assistant calls the GetPanel function with T of the type UserControl

Naming Convention

Plugin names should end with the word Plugin. For example if your plugin works with email then the proper plugin name would be EmailPlugin.
